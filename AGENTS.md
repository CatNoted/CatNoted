# AGENTS.md — CatNoted Monorepo Architecture & Agent System Guidelines

Welcome to **CatNoted** (`catnoted.com`), an AI-native spatial knowledge workspace combining **AFFiNE-style hybrid document/canvas editor**, **Obsidian-style knowledge graphs**, and **Space Agent client-side browser runtime**.

This document serves as the master specification, execution guidelines, and context definition for AI Agents and developers working inside this monorepo.

---

## 1. System Overview & Vision

CatNoted transforms static note-taking into a living, programmable workspace:

* **Hybrid Document & Canvas (AFFiNE-style):** Seamlessly transition between linear document view (`Doc Mode`) and infinite edgeless whiteboard (`Canvas Mode`) backed by a single CRDT data tree.
* **Knowledge Graph Engine (Obsidian-style):** Interactive 2D/3D force-directed node graph visualizing links (`[[backlinks]]`), tags, and dynamic relationships between pages, canvas shapes, and widgets.
* **Client-Side AI Agent Runtime (Space Agent-style):** A browser-executed AI agent operating on a local Virtual File System (VFS). The agent can dynamically compile, execute, and render interactive JavaScript widgets/tools directly inside the workspace without heavy server infrastructure.

```text
├── package.json               # Root Workspace Config
├── pnpm-workspace.yaml        # Workspace Package Definitions
└── turbo.json                 # Turborepo Build Pipelines

```

## 3. Package Responsibilities

### `apps/web`

* Client-facing web entry point for `catnoted.com` (Vite + React SPA).
* Manages UI layouts, modal dialogs, global state providers, and theme systems.
* Handles encrypted sync (E2EE) and Bring-Your-Own-Key (BYOK) settings.

### `packages/agent-runtime`

* **Virtual File System (VFS):** Operates an in-memory / IndexedDB file system storing user spaces, settings, and skills.
* **Sandboxed Execution Environment:** Runs dynamic JavaScript code generated by the agent inside an isolated iframe or Web Worker.
* **LLM Communication Loop:** Manages token-efficient system prompts, skill injection (`skills/*.md`), and transient message buffers.

### `packages/editor`

* Provides block-based document editing capabilities.
* Integrates CRDTs (e.g., Yjs / BlockSuite) for real-time state synchronization across document views.
* Embeds custom block extensions for live AI-rendered widgets.

### `packages/canvas`

* Implements infinite canvas spatial rendering (pan, zoom, multi-select, shape connections).
* Syncs document blocks seamlessly into spatial canvas cards.

### `packages/graph`

* Parses document references, `[[wiki-links]]`, and metadata across the VFS.
* Renders interactive node-edge graph views using Canvas/WebGL.

### `packages/shared`

* Central source of truth for interfaces: `BlockNode`, `CanvasElement`, `GraphEdge`, `WidgetSpec`, `AgentMessage`.

## 4. Agent Execution Guidelines & Protocols

When an AI Agent is modifying code or running inside the CatNoted runtime, it **must** adhere to the following rules:

### A. Client-Side First & Privacy Rule

1. **Zero Unnecessary Server Calls:** All agent operations, code execution, and data mutations must run within the client-side browser runtime (Web Workers / Sandboxed Iframes).
2. **Encrypted & Local Persistence:** User files and VFS nodes must persist locally via `IndexedDB` or `File System Access API`. Do not leak plain-text notes or API keys to external telemetry services.

### B. Dynamic Widget Generation Protocol

* Widgets created by the agent must be self-contained JavaScript/HTML/CSS modules rendered via the `agent-runtime` sandbox.
* The agent uses low-token YAML/plain-text responses to invoke widget rendering instead of verbose JSON payloads.
* UI widgets must automatically adopt CatNoted design tokens (CSS variables for dark/light themes).

### C. State Synchronization

* Never alter the Document DOM directly. Always mutate through the `packages/shared` state tree or Yjs CRDT providers.
* Updating a note **must** automatically update its corresponding node in `packages/graph` and its visual block in `packages/canvas`.

## 5. Development Commands & Conventions

### Workspaces Commands

* **Start Dev Server:** `pnpm dev`
* **Build All Packages:** `pnpm build`
* **Type-Check Codebase:** `pnpm typecheck`
* **Lint Codebase:** `pnpm lint`

### Conventions

* **Language:** TypeScript (Strict mode enabled across all packages).
* **Package Manager:** `pnpm` (Do NOT use `npm` or `yarn` directly).
* **Styling:** Tailwind CSS with CSS variables for dynamic theme switching.
* **Code Formatting:** Prettier + ESLint with monorepo rules.

---

## 6. How AI Assistants Should Contribute

If you are an AI coding assistant (Cursor, Claude Code, GPT) working on this codebase:

1. Always check `packages/shared` for existing types before creating new ones.
2. Maintain clean separation between workspace rendering (`editor`/`canvas`) and agent execution (`agent-runtime`).
3. Ensure all new components support both Light and Dark themes natively.
4. Keep functions modular and write clear inline comments for complex mathematical algorithms (e.g., graph layout engines, canvas viewport transforms).
