# ARCHITECTURE.md — CatNoted Technical Architecture & System Blueprint

This document defines the high-level system architecture, data models, component interactions, and execution flow for **CatNoted** (`catnoted.com`).

---

## 1. Architectural Core Principles

1. **Client-Side First (Zero-Knowledge Runtime):** All note rendering, canvas operations, knowledge indexing, and AI agent execution happen inside the user's browser runtime (VFS + Web Workers + Sandboxed IFrames).
2. **Single Source of Truth (Unified Data Tree):** Document Mode (linear blocks) and Canvas Mode (edgeless whiteboard) are **two visual views** driven by a single underlying Yjs CRDT (Conflict-free Replicated Data Type) tree.
3. **Reactive Graph Indexing:** The Obsidian-style Knowledge Graph continuously indexes references (`[[backlinks]]`), tags, and dynamic block relations reactively from the CRDT state tree.
4. **Sandboxed Agent Tooling:** The agent runtime generates and executes HTML/JS widgets within an isolated, sandboxed environment to prevent DOM contamination and security breaches.

## 3. Package Architecture & Responsibilities

### `apps/web` (Application Shell)

* **Framework:** Vite + React SPA.
* **Role:** The entry point application. Manages workspace layouts, sidebars, modal managers, global keyboard shortcuts (`Cmd+K`), theme states (Light/Dark), and encrypted storage configurations.

### `packages/shared` (Data Schemas & Types)

* **Role:** Central source of truth for TypeScript interfaces and schemas.
* **Key Exports:**
  * `BlockNode`: Core structural node (text, image, heading, AI widget container).
  * `CanvasElement`: Spatial attributes (x, y, scale, rotation, links).
  * `GraphNode` & `GraphEdge`: Graph structure definitions for WebGL/Canvas rendering.
  * `AgentMessage` & `WidgetSpec`: Protocols for communicating with the agent runtime.

### `packages/editor` (Document Engine)

* **Framework:** BlockSuite / Yjs CRDT integration.
* **Role:** Renders block-based documents sequentially. Emits CRDT delta events whenever blocks are created, reordered, or edited.

### `packages/canvas` (Spatial Whiteboard Engine)

* **Role:** Renders an infinite 2D viewport (pan, zoom, multi-select). Renders document blocks as spatial cards or connected shapes on the canvas. Uses coordinate matrix transforms for smooth rendering.

### `packages/graph` (Knowledge Graph Engine)

* **Framework:** D3.js / WebGL Force-Directed Graph Engine.
* **Role:** Scans all VFS nodes and CRDT blocks for `[[backlink]]` syntax, tags, and spatial card connections. Renders an interactive 2D/3D force graph in real-time.

### `packages/agent-runtime` (Space Agent Engine)

* **Role:** Client-side AI runtime.
* **Key Components:**
  * **VFS (Virtual File System):** In-memory and IndexedDB storage representing files, skills (`skills/*.md`), and space state.
  * **JS Sandbox:** Executes code generated by the agent inside a securely sandboxed Iframe via `postMessage` protocol.
  * **LLM Engine:** Interacts directly with user-provided API Keys (OpenAI, Anthropic, Gemini, Ollama) using optimized prompt buffers.

## 4. State Synchronization Flow

### 4.1 Local State Synchronization Flow

When a user or an AI Agent mutates a document locally, the change propagates reactively through the system:

```text
[ User Edit / AI Agent Action ]
             │
             ▼
   Mutates Yjs CRDT Tree  ──────► [ IndexedDB / VFS Persistence ]
             │
   ┌─────────┼────────────────────────┐
   │         │                        │
   ▼         ▼                        ▼
[ Editor ] [ Canvas ]           [ Graph Engine ]
 Re-renders  Updates Spatial      Re-computes Force Layout
 Block DOM   Card View            & Edge Connections
```

1. **Mutation:** Any user input or AI action triggers a transaction on the local Yjs document.
2. **Reactivity:** Yjs observers emit change events.
3. **View Update:**
   - `packages/editor` updates the active block line.
   - `packages/canvas` updates card positions and link arrows.
   - `packages/graph` re-computes layout forces and node connections.
4. **Persistence:** The local IndexedDB engine automatically commits CRDT deltas.

---

### 4.2 Cross-Platform Cloud Synchronization Flow (Supabase)

To sync data seamlessly across Web, Desktop (Tauri), and Mobile (Flutter/React Native), CatNoted uses a Zero-Knowledge Supabase backend:

```text
[ Web / Desktop / Mobile App ]
               │
               ├─► 1. User mutates document/canvas (Local Yjs State)
               ├─► 2. Client encrypts payload locally (AES-GCM via Web Crypto API)
               │
               ▼  (Sync encrypted payload via HTTPS / WebSocket)
+-------------------------------------------------------------+
| SUPABASE CLOUD BACKEND                                      |
|                                                             |
|  [ Supabase Auth ]     ──► Validates user session token     |
|  [ Postgres / Realtime ]──► Stores & broadcasts E2EE deltas |
|  [ Storage Buckets ]   ──► Stores encrypted media/files     |
+-------------------------------------------------------------+
               │
               ▼  (Realtime broadcast via WebSockets)
[ User's Other Devices (Mobile / Tablet / Desktop) ]
               │
               └─► Decrypts payload locally & merges into local Yjs Doc
```

1. **Client-Side Encryption:** Before leaving the device, CRDT deltas and assets are encrypted locally using AES-GCM (Zero-Knowledge Architecture).
2. **Blind Storage Transmission:** Encrypted payloads are sent to Supabase Postgres via `y-supabase` or WebSocket channels. Supabase servers cannot read the content.
3. **Real-time Broadcast:** Supabase Realtime notifies and pushes updates to all active sessions belonging to the user.
4. **Local Conflict Resolution:** Receiving devices decrypt the payload and Yjs automatically resolves any state conflicts via CRDT algorithm.

## 5. Agent Security & Sandbox Execution

To prevent malicious code or broken AI-generated JavaScript from freezing or compromising the app:

```text
+-------------------------------------------------------------------+
| MAIN THREAD (apps/web)                                            |
|                                                                   |
| [ User Space ] <---> [ Event Bus ]                                |
|                           │                                       |
|                  postMessage Protocol                             |
|                           │                                       |
|                           v                                       |
|  +-------------------------------------------------------------+  |
|  | SANDBOXED IFRAME (packages/agent-runtime)                  |  |
|  | Attributes: sandbox="allow-scripts"                         |  |
|  |                                                             |  |
|  | [ Compiled Widget JS ] ---> Renders UI Widget in Sandbox  |  |
|  +-------------------------------------------------------------+  |
+-------------------------------------------------------------------+
```

1. **Isolation:** AI-created widgets run inside an Iframe configured with `sandbox="allow-scripts"`.
2. **Communication:** Data passes between the main app and the widget using `window.postMessage` with strict schema validation.
3. **CSS Isolation:** Widgets inherit CatNoted CSS variables (design tokens) via a scoped Shadow DOM to maintain theme consistency without polluting the global stylesheet.

## 6. Storage & BYOK (Bring Your Own Key) Strategy

* **Local Storage Layer:** Primary persistence uses browser `IndexedDB` (via VFS) or native local file system APIs for maximum speed and offline support.
* **Backend Cloud Infrastructure (Supabase):**
  * **Auth:** Handles multi-device user authentication (OAuth, Passkeys, Magic Links).
  * **Database & Realtime:** Manages workspace metadata, access control (RLS), and E2EE document delta broadcasting.
  * **Storage Buckets:** Stores encrypted binary attachments (images, PDFs, audio).
* **API Key Management:** User-provided LLM keys (OpenAI, Anthropic, Gemini, Localhost Ollama) are stored encrypted in local storage and sent **directly to the LLM endpoints** from the browser. Keys never touch Supabase or CatNoted servers.
